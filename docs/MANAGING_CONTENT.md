# Руководство по управлению контентом (Архитектура на SQLite)

**Важное обновление:** Бэкенд админ-панели был мигрирован на NestJS, а фронтенд рефакторинг с использованием Zustand. Подробности см. в [14. Миграция на NestJS и рефакторинг админ-панели](src/Docs/14-backend-nestjs-migration.md).

## Обзор

Проект использует централизованную базу данных **SQLite** для хранения всего контента. Этот подход заменяет старую, файловую систему, основанную на `.md` файлах и сложных `JSON-манифестах`, и обеспечивает надежность, гибкость и простоту в управлении данными.

-   **Файл базы данных**: `src/data/database.db`
-   **Управление**: Локальный API-сервер (`scripts/admin-server.ts`) и админ-панель.

## 1. Архитектура данных

### 1.1. Схема базы данных

База данных состоит из двух основных таблиц:

1.  **`content`**: Хранит основную информацию о каждой единице контента.
    -   `id`: `INTEGER PRIMARY KEY` - Уникальный, неизменяемый числовой идентификатор.
    -   `slug`: `TEXT` - Уникальный URL-дружественный идентификатор (например, `dzhazovyy-dzhem-seyshn`).
    -   `title`: `TEXT` - Заголовок.
    -   `date`: `TEXT` - Дата в формате `YYYY-MM-DD`.
    -   `category`: `TEXT` - Основная категория (`новость`, `афиша`, `фотоальбом`, `дипломы`).
    -   `subcategory`: `TEXT` - Опциональная подкатегория (например, для фотоальбомов).
    -   `body`: `TEXT` - Основной текст статьи в формате Markdown.
    -   `cover_image_src`: `TEXT` - Путь к изображению-обложке.
    -   `linked_photoalbum_id`: `INTEGER` - ID фотоальбома, если контент (например, новость) связан с фотоальбомом.
    -   ...и другие поля (`time`, `venue`, `tags`).

2.  **`gallery_images`**: Хранит информацию об изображениях в галереях.
    -   `id`: `INTEGER PRIMARY KEY`.
    -   `content_id`: `INTEGER` - Внешний ключ, связывающий изображение с записью в таблице `content`.
    -   `src`: `TEXT` - Путь к файлу изображения.
    -   `title`: `TEXT` - Название изображения.

3.  **`videos`**: Хранит информацию о видеозаписях.
    -   `id`: `INTEGER PRIMARY KEY AUTOINCREMENT` - Уникальный идентификатор видео.
    -   `legacy_id`: `TEXT` - Опциональный идентификатор из старой системы.
    -   `title`: `TEXT NOT NULL` - Заголовок видео.
    -   `description`: `TEXT` - Описание видео.
    -   `video_url`: `TEXT NOT NULL` - URL видео (например, YouTube, RuTube).
    -   `date`: `TEXT NOT NULL` - Дата публикации видео в формате `YYYY-MM-DD`.
    -   `source`: `TEXT` - Источник видео (например, 'rutube', 'youtube').

### 1.2. Поток данных

#### Режим разработки

В режиме разработки (`npm run dev`) клиентская часть сайта обращается к локальному API-серверу для получения контента:

1.  **React-компонент** (например, `NewsPage.tsx`) вызывает функцию-загрузчик.
2.  **Функция-загрузчик** выполняет `fetch` запрос к локальному API-серверу (например, `fetch('http://localhost:4000/api/news')`).
3.  **API-сервер** (`scripts/admin-server.ts`) делает SQL-запрос к базе данных `database.db`.
4.  Сервер отдает данные в формате **JSON**, которые отображаются в компоненте.

#### Сборка и развертывание (Production)

Для развертывания сайта и обеспечения высокой производительности используется генерация статических JSON-файлов:

1.  Скрипт **`scripts/generate-static-data.ts`** читает данные из `database.db`.
2.  Он преобразует их в статические файлы (`public/data/news.json`, `public/data/afisha.json` и т.д.).
3.  Клиентская часть в продакшен-сборке запрашивает эти статические файлы напрямую (например, `fetch('/data/news.json')`).

Запуск `generate-static-data.ts` интегрирован в скрипты `npm run dev`, `npm run build`, `npm run admin` и `npm run deploy`, что гарантирует актуальность данных.

## 2. Управление контентом через админ-панель

Для создания, редактирования и удаления контента используется локальная админ-панель.

### 2.1. Запуск

Админ-панель доступна по адресу `http://localhost:5173/admin.html` после одновременного запуска сервера разработки и админ-сервера:

```bash
# В одном терминале
npm run dev

# В другом терминале
npm run admin
```

Админ-панель — это React-приложение, которое взаимодействует с API-сервером для всех операций с базой данных (CRUD).

### 2.2. Ключевые возможности и исправления

-   **Темный режим** по умолчанию.
-   **Корректное форматирование даты** в полях ввода.
-   **Отображение фотогалереи и библиотеки** из статических JSON файлов.
-   **Устранение дублирования фотоальбомов**: вкладка "Фотоальбомы" теперь показывает только самостоятельные альбомы.
-   **Исправление ошибок обновления контента** и улучшение UX.

### 2.3. API-эндпоинты

Сервер `admin-server.ts` предоставляет два типа эндпоинтов:

-   **Публичные (`/api/news`, `/api/afisha`, `/api/videos`)**: Используются сайтом для получения контента (например, `/api/videos` для получения списка видео).
-   **Административные (`/api/content`, `/api/content/:id`, `/api/videos`, `/api/videos/:id`)**: Используются админ-панелью для CRUD-операций.
    -   `/api/content`, `/api/content/:id`: Для новостей, афиш и других общих сущностей.
    -   `/api/videos`, `/api/videos/:id`: Для создания, просмотра, редактирования и удаления видеозаписей.
        -   `GET /api/videos`: Получить все видео.
        -   `GET /api/videos/:id`: Получить видео по ID.
        -   `POST /api/videos`: Создать новое видео.
        -   `PUT /api/videos/:id`: Обновить существующее видео.
        -   `DELETE /api/videos/:id`: Удалить видео по ID.

## 3. Скрипты для работы с базой данных

В `package.json` доступны скрипты для управления базой данных:

-   `npm run db:setup`: Создает файл `database.db` и пустые таблицы.
-   `npm run db:cleanup`: Стандартизирует значения в колонке `category`.
-   `npm run db:add-subcategory`: Добавляет колонку `subcategory` в таблицу `content`.
-   `npm run db:migrate`: (Выполнялся один раз) Переносит контент из старой файловой системы в базу данных SQLite.

Эта система обеспечивает надежное и профессиональное управление контентом проекта.

## 4. Возможности Markdown и улучшенное отображение контента

Для повышения гибкости в представлении текстового контента и улучшения пользовательского опыта были внесены следующие изменения:

### 4.1. Использование Markdown в контенте

Текстовые поля для **новостей** и **афиш** теперь поддерживают синтаксис Markdown. Это позволяет:
-   Автоматически преобразовывать URL-адреса в кликабельные ссылки.
-   Форматировать текст (полужирный, курсив и т.д.).
-   Корректно отображать переносы строк.

**Как создавать ссылки:**
*   **Встроенные ссылки:** `[Текст ссылки](URL)` (например, `[наш сайт](https://jazzcollege48.ru)`)
*   **Автоматические ссылки:** Просто вставьте URL (`https://vk.com/album-215322611_286149512`)

**Как делать переносы строк:**
*   **Новый абзац:** Оставьте **одну пустую строку** между блоками текста.
*   **Принудительный перенос строки (без нового абзаца):** Поставьте **два или более пробелов** в конце строки, затем нажмите Enter.
*   **HTML тег:** Используйте `<br>` для принудительного переноса.

**Затронутые компоненты:** `NewsDetailPage`, `NewsPage`, `NewsPreview` и `Concerts` теперь используют `react-markdown` для рендеринга контента.

### 4.2. Улучшения визуальных элементов

*   **Интерактивные обложки:** При наведении мыши на обложки карточек (новости, афиши, детальные страницы) теперь отображается полупрозрачный оверлей с "лупой", указывающий на возможность открытия изображения в увеличенном виде. Курсор мыши при этом меняется на `pointer`, подчеркивая интерактивность.
*   **Единая высота текстовых блоков:** Для улучшения визуальной симметрии и выравнивания контента, текстовые блоки в карточках новостей и видео (заголовки и описания) теперь имеют фиксированное количество строк благодаря использованию свойства `line-clamp` и CSS Flexbox. Это обеспечивает единообразный вид всех карточек, независимо от длины текста.

## 5. Развертывание SPA и решение проблемы 404

При развертывании одностраничных приложений (SPA) на веб-серверах (например, Apache на `reg.ru`) часто возникает ошибка 404 "Page not found" при прямом переходе на внутренние маршруты (например, `/news`) или обновлении страницы. Это происходит потому, что сервер ищет физический файл по этому адресу, а не отдает `index.html`.

Для решения этой проблемы необходимо настроить веб-сервер.

### 5.1. Конфигурация для Apache (.htaccess)

Если ваш хостинг использует Apache (что часто бывает на `reg.ru`), создайте (или измените) файл `.htaccess` в корневой директории вашего развернутого приложения (там же, где `index.html`) со следующим содержимым:

```apache
Options -MultiViews
RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule ^ index.html [QSA,L]
```

Этот файл гарантирует, что любой запрос, не соответствующий существующему файлу или директории, будет перенаправлен на `index.html`, позволяя React Router обрабатывать маршрутизацию на стороне клиента.

**Автоматизация:** Файл `.htaccess` теперь автоматически включается в сборку проекта (`npm run build`) путем его размещения в директории `public/.htaccess`, и будет загружен на FTP вместе с остальными файлами.

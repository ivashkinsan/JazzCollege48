# 10. Реализация хранения данных на SQLite

## 1. Введение
Данный документ описывает реализованную в проекте архитектуру хранения данных, основанную на **SQLite**. Эта система была внедрена для централизованного и структурированного хранения всего динамического контента сайта (новости, афиши и т.д.), заменив собой предыдущую файловую систему.

## 2. Мотивация
*   **Надежность:** Устранение ошибок и хрупкости, связанных с парсингом файлов и синхронизацией манифестов.
*   **Гибкость:** Возможность легкого редактирования всех полей контента, включая заголовок и дату, без необходимости переименовывать файлы и папки.
*   **Централизация:** Весь контент хранится в одном месте (`src/data/database.db`), что упрощает резервное копирование и управление.
*   **Производительность:** SQL-запросы работают значительно быстрее, чем чтение и парсинг множества файлов с диска.

## 3. Выбор СУБД
В качестве СУБД была выбрана **SQLite** через библиотеку `better-sqlite3`.
*   **Причина выбора:** SQLite — это легковесная, встраиваемая база данных, которая не требует отдельного серверного процесса. Вся база данных хранится в одном файле (`src/data/database.db`), что идеально подходит для данного проекта, так как это сохраняет его самодостаточность и упрощает развертывание.

## 4. Схема базы данных (реализованная)

### Таблица `content`
Хранит основную информацию о каждой единице контента.
```sql
CREATE TABLE IF NOT EXISTS content (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    slug TEXT UNIQUE NOT NULL,
    title TEXT NOT NULL,
    date TEXT NOT NULL,
    category TEXT NOT NULL,
    subcategory TEXT,
    time TEXT,
    venue TEXT,
    tags TEXT,
    body TEXT NOT NULL,
    cover_image_src TEXT
);
```

### Таблица `gallery_images`
Хранит информацию об изображениях в галереях.
```sql
CREATE TABLE IF NOT EXISTS gallery_images (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content_id INTEGER NOT NULL,
    src TEXT NOT NULL,
    title TEXT,
    FOREIGN KEY (content_id) REFERENCES content (id) ON DELETE CASCADE
);
```

### Таблица `achievements`
Хранит информацию о достижениях (дипломах, наградах).
```sql
CREATE TABLE IF NOT EXISTS achievements (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    legacy_id TEXT,
    title TEXT NOT NULL,
    student_name TEXT,
    competition TEXT NOT NULL,
    date TEXT NOT NULL,
    place TEXT,
    city TEXT, -- Город, где проходило событие
    category TEXT,
    image_src TEXT
);
```

## 5. Интеграция с приложением

### 5.1. API-сервер
Вся логика работы с базой данных инкапсулирована в локальном API-сервере `scripts/admin-server.ts`. Этот сервер на Node.js/Express предоставляет эндпоинты как для админ-панели (чтение и запись), так и для публичной части сайта (только чтение).

### 5.2. Фронтенд
React-приложение (и админ-панель, и основной сайт) **никогда не обращается к базе данных напрямую**. Оно взаимодействует исключительно с API-сервером через `fetch`-запросы, что обеспечивает безопасность и четкое разделение логики.

## 6. Скрипты управления
Для удобства работы с базой данных в `package.json` были добавлены следующие скрипты:
-   `npm run db:setup`: Создает файл БД и таблицы.
-   `npm run db:migrate`: Выполняет миграцию данных из старой файловой системы.
-   `npm run db:cleanup`: Стандартизирует категории контента.
-   `npm run db:add-subcategory`: Модифицирует схему, добавляя новые колонки.

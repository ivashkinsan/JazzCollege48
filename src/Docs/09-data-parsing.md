# 9. Парсинг и загрузка контента

## Обзор

В проекте существует две параллельные системы для работы с контентом:
1.  **Динамический контент (Новости, Афиша)**: Загружается во время работы приложения из `.md` файлов. Управляется скриптом `add-content.js`.
2.  **Статический медиа-манифест (Фотогалерея)**: Используется для страницы "Фотогалерея" и для связи `id` изображений с их путями.

---

## 1. Динамический контент (Новости, Афиша)

Эта система позволяет добавлять контент без пересборки проекта.

### 1.1. Хранение контента

-   **Расположение**: Весь контент (новости и афиши) хранится в `public/media/`.
-   **Структура**: Каждый материал (событие, новость) находится в своей папке, например, `public/media/2026/2026-01-26-my-event/`.
-   **Содержимое папки**: Внутри находится `.md` файл с текстом и метаданными, а также все относящиеся к нему изображения (`cover.jpg`, `gallery-1.jpg` и т.д.).

### 1.2. Механизм загрузки (`fetch` + `manifest.json`)

Вместо `import.meta.glob` (который работает только с `src/` и на этапе сборки), используется более гибкий подход:

1.  **Манифест (`public/media/manifest.json`)**: Это ключевой файл, содержащий JSON-массив с путями ко всем `.md` файлам.
    ```json
    [
      "/media/2026/.../event-one.md",
      "/media/2026/.../event-two.md"
    ]
    ```
    Этот файл автоматически обновляется скриптом `scripts/add-content.js`.

2.  **Загрузчики (`newsLoader.ts`, `afishaLoader.ts`)**:
    - При вызове (например, `loadNews()`) функция сначала загружает (`fetch`) `/media/manifest.json`.
    - Затем она итерирует по списку путей и асинхронно загружает (`fetch`) содержимое каждого `.md` файла.
    - Каждый файл парсится функцией `parseMarkdown`.

### 1.3. Функция `parseMarkdown(content: string)`

Эта функция является сердцем механизма парсинга `.md` файлов. Она принимает строковое содержимое файла и возвращает объект с двумя полями: `frontmatter` и `body`.

-   **`frontmatter`**: Объект с метаданными (Title, Date, Category и т.д.).
-   **`body`**: Текстовое содержимое после блока frontmatter.

**Важные детали парсера:**
-   Он корректно обрабатывает BOM (`\uFEFF`) в начале файла.
-   Он справляется с разными окончаниями строк (`\n` и `\r\n`).
-   Он извлекает значения из frontmatter, удаляя внешние кавычки.

## 2. Статический медиа-манифест (`src/data/media-manifest.json`)

Этот файл служит другой цели и **не связан** с `public/media/manifest.json`.

### 2.1. Назначение

1.  **Источник данных для Фотогалереи**: Страница `/photos` читает этот файл напрямую, чтобы отобразить список фотоальбомов.
2.  **Связующее звено для изображений**: Загрузчики (`newsLoader.ts`, `afishaLoader.ts`) используют этот файл, чтобы по `coverImageId` или `id` из галереи найти реальный путь (`src`) к изображению.

### 2.2. Структура

Это массив объектов `PhotoAlbum`, где каждый альбом имеет:
-   `albumId`, `albumTitle`, `albumDate`, `albumCategory`
-   Массив `photos`, где у каждой фотографии есть `id`, `src`, и `title`.

```json
[
  {
    "albumId": "master-class-rakhmaninov-2024",
    "albumTitle": "Мастер-класс...",
    "albumDate": "2024-02-06",
    "albumCategory": "мастер-классы",
    "photos": [
      {
        "id": "mc_rakhmaninov_2024_01",
        "src": "/media/2024/.../cover.jpg",
        "title": "Общее фото"
      },
      // ...
    ]
  }
]
```

### 2.3. Обновление

Этот файл также автоматически обновляется скриптом `scripts/add-content.js`.

## 3. Функции `parseGalleryIds` и `stripGallery`

-   **`parseGalleryIds(body)`**: Эта функция извлекает **идентификаторы** (не пути!) из блока `<!-- gallery -->` в `.md` файле. Например:
    ```markdown
    <!-- gallery -->
    - mc_rakhmaninov_2024_02
    - mc_rakhmaninov_2024_03
    ```
    Загрузчик затем использует эти `id`, чтобы найти полные объекты фотографий в `media-manifest.json`.

-   **`stripGallery(body)`**: Удаляет блок `<!-- gallery -->` из контента, чтобы он не отображался в тексте новости.

Эта двухуровневая система (динамический манифест для контента + статический для медиа) позволяет гибко управлять данными.

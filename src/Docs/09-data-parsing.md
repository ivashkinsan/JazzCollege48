# 9. Хранение и загрузка контента (Архитектура на SQLite)

## Обзор

Проект использует централизованную базу данных **SQLite** для хранения всего контента. Это заменяет старую, файловую систему, основанную на `.md` файлах и сложных `JSON-манифестах`. Такой подход обеспечивает надежность, гибкость и простоту в управлении данными.

-   **Файл базы данных**: `src/data/database.db`
-   **Управление**: Локальный API-сервер (`scripts/admin-server.ts`)

---

## 1. Архитектура данных

### 1.1. Схема базы данных

База данных состоит из двух основных таблиц:

1.  **`content`**: Хранит основную информацию о каждой единице контента.
    -   `id`: `INTEGER PRIMARY KEY` - Уникальный, неизменяемый числовой идентификатор.
    -   `slug`: `TEXT` - Уникальный URL-дружественный идентификатор (например, `dzhazovyy-dzhem-seyshn`).
    -   `title`: `TEXT` - Заголовок.
    -   `date`: `TEXT` - Дата в формате `YYYY-MM-DD`.
    -   `category`: `TEXT` - Основная категория (`новость`, `афиша`, `фотоальбом`, `дипломы`).
    -   `subcategory`: `TEXT` - Опциональная подкатегория (например, для фотоальбомов).
    -   `body`: `TEXT` - Основной текст статьи в формате Markdown.
    -   `cover_image_src`: `TEXT` - Путь к изображению-обложке.
    -   `linked_photoalbum_id`: `INTEGER` - ID фотоальбома, если контент (например, новость) связан с фотоальбомом.
    -   ...и другие поля (`time`, `venue`, `tags`).

2.  **`gallery_images`**: Хранит информацию об изображениях в галереях.
    -   `id`: `INTEGER PRIMARY KEY`.
    -   `content_id`: `INTEGER` - Внешний ключ, связывающий изображение с записью в таблице `content`.
    -   `src`: `TEXT` - Путь к файлу изображения.
    -   `title`: `TEXT` - Название изображения.

### 1.2. Поток данных для сайта (режим разработки)

В режиме разработки (`npm run dev`) клиентская часть сайта обращается к локальному API-серверу для получения контента:

1.  **React-компонент** (например, `NewsPage.tsx`) вызывает функцию-загрузчик (например, `loadNews()`).
2.  **Функция-загрузчик** выполняет сетевой запрос (`fetch`) к локальному API-серверу (например, `fetch('http://localhost:4000/api/news')`).
3.  **API-сервер** (`scripts/admin-server.ts`), получив запрос, выполняет SQL-запрос к базе данных `database.db`.
4.  Сервер отдает запрошенные данные в формате **JSON**.
5.  Функция-загрузчик получает JSON и передает его React-компоненту для отображения.

Эта архитектура обеспечивает четкое разделение между клиентской и серверной логикой в процессе разработки.

### 1.3. Генерация статических данных для развертывания сайта

Для развертывания сайта и обеспечения высокой производительности и независимости от API-сервера используется процесс генерации статических JSON-файлов:

1.  Скрипт **`scripts/generate-static-data.ts`** читает данные непосредственно из базы данных `database.db`.
2.  Он преобразует эти данные в статические JSON-файлы (например, `public/data/news.json`, `public/data/afisha.json`, `public/data/library.json`, `public/data/photoalbums.json`).
3.  Эти статические файлы затем используются клиентской частью сайта (например, `NewsPage.tsx`, `AfishaPage.tsx`, `LibraryPage.tsx`, `PhotosPage.tsx`) путем прямого `fetch` запроса к ним (например, `/data/news.json`).

Запуск `generate-static-data.ts` интегрирован в следующие скрипты:
-   `npm run dev`: Запускается перед стартом сервера разработки, чтобы обеспечить актуальность статических данных.
-   `npm run build`: Запускается перед сборкой проекта для продакшена.
-   `npm run admin`: Запускается перед стартом админ-сервера.
-   `npm run deploy`: Запускается как часть процесса сборки для развертывания.

Таким образом, статические JSON-файлы всегда актуальны и доступны для сайта в продакшен-среде.

## 2. Управление контентом

### 2.1. Админ-панель

Для управления контентом используется локальная админ-панель, доступная по адресу `http://localhost:5173/admin.html` (после запуска `npm run admin` и `npm run dev`). Админ-панель — это React-приложение, которое использует API-эндпоинты для всех операций (CRUD).

**Ключевые улучшения и исправления:**

*   **Темный режим по умолчанию:** Админ-панель теперь загружается в темном режиме по умолчанию, используя атрибут `data-theme="dark"` и CSS-переменные для адаптации стилей.
*   **Отображение фотогалереи и библиотеки:** Исправлено отображение данных фотогалереи и библиотеки на основном сайте (в продакшен-сборке) путем переключения на использование статических JSON-файлов вместо локального API.
*   **Устранение дублирования фотоальбомов:** Вкладка "Фотоальбомы" теперь отображает только "самостоятельные" альбомы, исключая те, которые связаны с новостями через `linked_photoalbum_id`, что устраняет дублирование.
*   **Исправление ошибки обновления контента:** Устранена ошибка `RangeError: Too many parameter values were provided` (ранее `SQLITE_CONSTRAINT: NOT NULL constraint failed: content.subcategory`) при обновлении контента, путем корректной передачи параметра `subcategory` и обеспечения правильного форматирования параметра `category` как одиночной строки.
*   **Корректное отображение связанного фотоальбома:** В выпадающем списке "Связанный фотоальбом" теперь корректно отображается заголовок выбранного фотоальбома благодаря унификации типов данных (строка) для `linked_photoalbum_id` во всех связанных функциях React-компонента.
*   **Улучшенное превью новостей:** В превью новостей теперь более надежно отображается кнопка "Читать далее", если полный текст (`content`) длиннее сокращенного описания (`description`).

Благодаря базе данных, все поля, включая `title`, `date` и `slug`, теперь можно свободно редактировать.

### 2.2. API-эндпоинты

Сервер `admin-server.ts` предоставляет два типа эндпоинтов:

-   **Публичные (`/api/news`, `/api/afisha`)**: Используются основным сайтом для получения отформатированного контента.
-   **Административные (`/api/content`, `/api/content/:id`)**: Используются админ-панелью для создания, чтения, обновления и удаления записей.

## 3. Скрипты для работы с БД

В `package.json` добавлены скрипты для управления базой данных:

-   `npm run db:setup`: Создает файл `database.db` и пустые таблицы, если их нет.
-   `npm run db:cleanup`: Выполняет стандартизацию категорий на основе жестко заданных списков.
-   `npm run db:add-subcategory`: Обновляет схему таблицы `content`, добавляя колонку `subcategory`.
-   `npm run db:migrate`: (Выполнялся один раз) Скрипт для переноса контента из старой файловой системы в базу данных SQLite.

Эта система обеспечивает надежное и профессиональное управление контентом проекта.

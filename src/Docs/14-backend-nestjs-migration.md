# 14. Миграция на NestJS и рефакторинг админ-панели

Этот документ описывает масштабные изменения, внесенные в архитектуру бэкенда и фронтенда админ-панели с целью повышения стабильности, поддерживаемости и масштабируемости проекта.

## 1. Миграция бэкенда на NestJS

**До:** Бэкенд админ-панели был реализован на простом Express.js сервере, где бизнес-логика была сильно переплетена с маршрутизацией, что создавало проблемы с читаемостью и тестируемостью.

**После:** Бэкенд был полностью переведен на фреймворк NestJS, который предоставляет мощную, модульную и строго типизированную архитектуру.

### Ключевые изменения:

*   **Установка NestJS:** Установлены основные зависимости NestJS (`@nestjs/core`, `@nestjs/common`, `@nestjs/platform-express`, `reflect-metadata`) и CLI (`@nestjs/cli`).
*   **Структура проекта:**
    *   Создана новая директория `src/server` для размещения всего кода бэкенда.
    *   Реализована стандартная структура NestJS:
        *   `src/server/main.ts`: Точка входа приложения.
        *   `src/server/app.module.ts`: Корневой модуль приложения.
        *   `src/server/app.controller.ts`, `src/server/app.service.ts`: Базовые компоненты приложения.
*   **Модули функциональности:** Для каждой основной сущности (Content, Graduates, Library) были созданы отдельные NestJS модули, включающие:
    *   **Service (`*.service.ts`):** Содержит всю бизнес-логику и взаимодействие с базой данных (CRUD-операции, файловые операции).
    *   **Controller (`*.controller.ts`):** Обрабатывает HTTP-запросы, используя декораторы NestJS (`@Get`, `@Post`, `@Delete`, `@Param`, `@Body`, `@UploadedFile`, `@UploadedFiles`, `@UseInterceptors`) для определения маршрутов и обработки данных.
    *   **Module (`*.module.ts`):** Объединяет сервисы и контроллеры, а также управляет зависимостями.
*   **DatabaseModule:** Создан специальный модуль `src/server/database/`, который предоставляет единый экземпляр соединения с базой данных `better-sqlite3` для инъекции в сервисы.
*   **Утилиты:** Общие утилиты (`utils.ts`, `generate-static-data.ts`) были перемещены из `scripts/api/` в `src/server/utils/` для лучшей инкапсуляции и соответствия архитектуре бэкенда.
*   **Обработка ошибок и валидация:** В сервисах добавлены `NotFoundException` и `BadRequestException` для более гранулярной обработки ошибок, возвращаемых API.
*   **Мультипарт-формы:** Для загрузки файлов (изображений) вместо `multer` напрямую теперь используются встроенные в NestJS интерцепторы (`FileInterceptor`, `FileFieldsInterceptor`).

### Обновление скриптов и конфигурации:

*   **`package.json`:**
    *   Скрипт `admin` теперь запускает NestJS сервер через `node dist/src/server/main.js` после компиляции.
    *   Добавлен скрипт `build:server` (`tsc -p tsconfig.node.json`) для компиляции TypeScript бэкенда.
    *   Скрипты `dev` и `build` обновлены для использования `tsx src/server/utils/generate-static-data.ts` для генерации статических данных.
    *   Все `db:*` скрипты теперь используют `tsx` вместо `ts-node`.
*   **`tsconfig.node.json`:** Обновлен для корректной компиляции NestJS (включены `emitDecoratorMetadata`, `experimentalDecorators`, исключены старые скрипты, добавлен `src/server/**/*.ts` и `src/types/**/*.ts`).
*   **`src/server/*/`:** Исправлены пути импорта (`.ts` на `.js` для скомпилированного кода) и корректное использование `__dirname` для ES-модулей.
*   **Удаление старого бэкенда:** Директория `scripts/api/` и файл `scripts/admin-server.ts` были удалены.

## 2. Рефакторинг фронтенда админ-панели с использованием Zustand

**До:** Фронтенд админ-панели (`src/admin/AdminApp.tsx`) был "компонентом-богом" (God Component) с большим количеством `useState` хуков, что усложняло управление состоянием и отладку.

**После:** Внедрена библиотека Zustand для централизованного управления состоянием.

### Ключевые изменения:

*   **Установка Zustand:** Установлена библиотека `zustand`.
*   **`src/admin/store.ts`:** Создан файл для определения глобального хранилища Zustand, которое теперь управляет такими состояниями, как `activeTab`, `mode`, `editingId`, `items`, `formData`, `status`, `isSubmitting`, `selectedFiles`.
*   **`src/admin/AdminApp.tsx`:** Основной компонент `AdminApp` был значительно упрощен. Теперь он получает состояние и функции действий из хранилища Zustand, становясь по большей части компонентом-представлением.
*   **Улучшенная логика выборки:** Исправлена логика определения параметра `manager` в `store.ts` для метода `fetchItems`, что позволило корректно запрашивать данные для всех вкладок, включая "Фотоальбомы", "Новости" и "Афиши".

## 3. Решение выявленных проблем

В процессе миграции и рефакторинга были решены следующие основные проблемы:

*   **Ошибки компиляции TypeScript:** Исправлены проблемы с путями импорта (`.ts` vs `.js`), отсутствием `emitDecoratorMetadata`, `experimentalDecorators` и неправильной конфигурацией `tsconfig`.
*   **Ошибки выполнения Node.js:** Устранена ошибка `ReferenceError: __dirname is not defined` путем использования стандартного для ES-модулей способа получения `__dirname`.
*   **`EADDRINUSE`:** Проблема занятости порта 4000 решалась принудительным завершением зависших процессов.
*   **Некорректное отображение данных:** Исправлена логика формирования запросов к базе данных в `ContentService.getAdminList` и логика выборки данных на фронтенде, что позволило всем вкладкам админ-панели корректно отображать данные.

## 4. Тестирование

Для обеспечения стабильности и надежности нового бэкенда на NestJS была настроена тестовая среда с использованием Jest.

### 4.1. Настройка и конфигурация

*   **Установка зависимостей:** Были установлены `jest`, `@types/jest`, `ts-jest`, `@nestjs/testing` и `supertest`.
*   **Конфигурация Jest:**
    *   `jest.config.js`: Основной конфигурационный файл для юнит-тестов. Он настроен для поиска файлов `*.spec.ts` в директории `src/`, использования `ts-jest` для трансформации TypeScript и сбора отчетов о покрытии.
    *   `jest.e2e.config.js`: Отдельный конфигурационный файл для E2E-тестов, который ищет файлы `*.e2e-spec.ts` в директории `test/`.
*   **Конфигурация TypeScript:** В `tsconfig.node.json` были добавлены типы для `jest` и `node` для корректной работы в тестовой среде.

### 4.2. Юнит-тесты

*   **Цель:** Проверка бизнес-логики в сервисах (`*.service.ts`) в изоляции.
*   **Расположение:** Тестовые файлы (`*.spec.ts`) находятся рядом с тестируемыми файлами.
*   **Мокирование:**
    *   Зависимости, такие как соединение с базой данных (`DATABASE_CONNECTION`), `fs/promises` и утилиты (`generateStaticData`), мокируются с помощью `jest.fn()` и `jest.mock()`.
    *   Это позволяет тестировать логику сервиса, не затрагивая реальную базу данных или файловую систему.
*   **Примеры:** Были написаны юнит-тесты для `ContentService`, `GraduatesService` и `LibraryService`.

### 4.3. E2E-тесты (End-to-End)

*   **Цель:** Проверка полного цикла запроса/ответа через HTTP, от контроллера до сервиса и обратно.
*   **Расположение:** Тесты находятся в директории `test/e2e/`.
*   **Инструменты:**
    *   `@nestjs/testing` используется для создания полного экземпляра приложения NestJS в тестовой среде.
    *   `supertest` используется для отправки HTTP-запросов к запущенному приложению и проверки ответов.
*   **Изоляция тестов:**
    *   E2E-тесты запускаются с `process.env.NODE_ENV = 'test'`.
    *   `DatabaseModule` настроен на использование отдельной тестовой базы данных (`test.database.db`), которая создается перед запуском тестов и удаляется после их завершения.
*   **Пример:** Был написан E2E-тест для `ContentController`, проверяющий GET-запросы.

### 4.4. Запуск тестов

В `package.json` добавлены следующие скрипты:

*   `npm test`: Запускает юнит-тесты.
*   `npm run test:e2e`: Запускает E2E-тесты.

Эти изменения значительно улучшают внутреннюю структуру проекта, делая его более стабильным, производительным и удобным для дальнейшей разработки и поддержки.

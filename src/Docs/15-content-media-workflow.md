# Документация: Рабочий процесс управления контентом и медиа

Этот документ описывает полный цикл создания и управления контентом (новости, фотоальбомы и т.д.), включая загрузку изображений, их сохранение и отображение.

## 1. Обзор архитектуры

Система управления контентом разделена на три основных части:

1.  **Административная панель (React)**: UI для создания, редактирования и удаления записей. Находится в `src/admin`.
2.  **Бэкенд-сервер (NestJS)**: API для обработки запросов от админ-панели, взаимодействия с базой данных и управления файлами. Находится в `src/server`.
3.  **Генератор статических данных**: Скрипт, который создает статические JSON-файлы для использования на основном сайте. Находится в `scripts/generate-static-data.ts`.

## 2. Процесс добавления/редактирования контента

### Шаг 1: Админ-панель (Клиент)

-   Когда пользователь заполняет форму и нажимает "Сохранить", срабатывает функция `submitForm` в `src/admin/store.ts`.
-   Эта функция собирает все данные из формы, включая текстовые поля и выбранные файлы, в объект `FormData`.
-   Загруженные изображения временно хранятся в стейте `selectedFiles` и отображаются в интерфейсе с помощью `URL.createObjectURL()`, что создает временные `blob:` ссылки.
-   `FormData` отправляется на сервер методом `POST` на соответствующий эндпоинт (например, `/api/content` или `/api/content/:id`).

### Шаг 2: Сервер (NestJS)

-   Запрос принимает `ContentController` в `src/server/content/content.controller.ts`.
-   **`FileFieldsInterceptor`**: Этот interceptor от NestJS (с использованием `multer`) обрабатывает `multipart/form-data` запросы. Он извлекает файлы из запроса и сохраняет их во временную папку (например, `tmp/uploads`). Информация о файлах (включая путь к временному файлу) передается в метод контроллера.
-   **`ContentService`**: Метод контроллера вызывает соответствующий метод в `src/server/content/content.service.ts` (`createContent` или `updateContent`).

#### Логика сохранения файлов в `ContentService`:

1.  **Формирование пути**:
    -   Для каждой записи создается уникальная папка на основе года, даты и "слаг" заголовка.
    -   `MEDIA_TARGET_ROOT`: Корневая папка для всех медиафайлов, путь которой `path.resolve(__dirname, '../../../public/media')`. **Важно:** Этот путь был исправлен, чтобы корректно указывать на папку `public/media` внутри проекта.
    -   Итоговый путь к папке: `public/media/{год}/{дата}-{слаг}/`.

2.  **Перемещение файла**:
    -   Файл перемещается из временной папки (`tmp/uploads/...`) в постоянную папку, созданную на предыдущем шаге.
    -   Для этого используется `fs.rename()`, что является атомарной и надежной операцией. Ранее использовалась связка `copyFile` + `unlink`, что было менее надежно.

3.  **Запись в базу данных**:
    -   **Только после успешного перемещения файла** в базу данных (`gallery_images`) добавляется запись о новом изображении.
    -   Путь (`src`), сохраняемый в БД, является относительным и всегда начинается с `/media/...`, что позволяет корректно отображать его на сайте.
    -   Реализована проверка на дубликаты, чтобы не добавлять в БД запись об изображении, которое уже существует в галерее для данной записи.

### Шаг 3: Генерация статических данных

-   После любой успешной операции CUD (Create, Update, Delete) в `ContentService` вызывается функция `generateStaticData()`.
-   Этот скрипт (`scripts/generate-static-data.ts`) заново читает данные из базы данных SQLite и пересоздает JSON-файлы в папке `public/data` (например, `news.json`, `photoalbums.json`).
-   Основной сайт использует эти статические JSON-файлы для отображения контента, что обеспечивает высокую производительность.

## 3. Утилиты и отладка

-   **Очистка "битых" ссылок**: Был создан скрипт `scripts/db-cleanup-images.ts`. Он проверяет каждую запись в `gallery_images` и удаляет те, которые ссылаются на несуществующие в файловой системе изображения. Это помогает поддерживать целостность данных.
-   **Логирование**: В `content.service.ts` добавлено расширенное логирование процесса перемещения файлов для быстрой диагностики проблем.
